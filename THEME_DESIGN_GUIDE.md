# 화면 모드 & 접근성 디자인 가이드

이 문서는 `src/index.css`, `src/contexts/ThemeContext.jsx`, `tailwind.config.js`에 구현된
테마/접근성 시스템의 설계 원칙을 정리한다. 새 컴포넌트를 만들거나 색을 추가할 때,
혹은 브랜드 컬러를 바꿀 때 이 문서를 기준으로 판단한다.

## 1. 토큰 아키텍처

색은 컴포넌트에서 직접 hex를 쓰지 않고, 전부 CSS 커스텀 프로퍼티(`--color-*`) →
Tailwind 시맨틱 클래스 2단계를 거친다.

```
src/index.css (:root, [data-theme='dark'], [data-theme='high-contrast'])
  --color-accent, --color-ink, --color-success ...
        ↓
tailwind.config.js (theme.extend.colors)
  accent: 'var(--color-accent)', ink: 'var(--color-ink)' ...
        ↓
컴포넌트 className="bg-accent text-ink"
```

**규칙: 컴포넌트는 `bg-stone-500`, `#3a7a2b` 같은 원색 클래스나 리터럴 hex를 절대
쓰지 않는다.** 항상 시맨틱 이름(`surface`, `ink`, `accent`, `success`, `danger`, `warning`,
`highlight`, `line`, `disabled` 등)을 쓴다. 그래야 테마 전환 시 값만 자동으로
바뀐다. 예외는 카카오 로그인 버튼(`#FEE500`)처럼 서드파티 브랜드 규정색뿐이다.

### 토큰 목록

| 토큰 | 용도 |
|---|---|
| `surface` / `surface-alt` / `surface-sunken` | 배경 / 카드·모달 배경 / 완료·비활성 등 눌린 배경 |
| `ink` / `ink-muted` / `ink-faint` | 본문 텍스트 / 보조 텍스트 / 장식적 요소(체브론 등) |
| `line` / `line-input` | 장식용 구분선(대비 기준 없음) / 입력창·인터랙티브 요소 테두리(3:1 이상) |
| `accent` / `accent-strong` / `accent-soft` / `accent-on` | 브랜드색 기본 / hover / hover 배경 워시 / 액센트 위에 올라가는 텍스트 |
| `highlight` | 보조 강조색(현재 캐러멜) — accent와 독립, 통계 카드 2번째 항목 등에 사용 |
| `success` / `danger` / `warning` / `warning-soft` | 상태색. success는 항상 `accent`(-strong)를 참조해 자동으로 따라감 |
| `disabled` / `disabled-ink` | 비활성 상태 |
| `focus` (`--color-focus-ring`) | 포커스 아웃라인. 항상 `accent`를 참조 |

## 2. 네 가지 화면 모드와 설계 원칙

라디오로 택1: **라이트 / 다크 / 시스템 설정 따르기 / 고대비**. `system`은 저장되는
"사용자 선호"일 뿐, 실제 렌더링 시점에 `<html data-theme>`은 항상 `light` | `dark` |
`high-contrast` 셋 중 하나로 해석돼서 CSS는 3개 케이스만 다루면 된다
(`ThemeContext.jsx`의 `resolveTheme()`).

### 원칙 A — 색상군 유지 (Material Design 3 톤 팔레트 방식)

**대비 단계가 달라져도 브랜드 색상군(hue)은 유지하고, 명도/채도만 조정한다.**
다크 모드의 accent(`#69b158`)와 고대비 모드의 accent(`#6ee7a0`)는 전부 라이트 모드
accent(`#3a7a2b`)와 같은 초록 계열(H≈109)이고, 명도만 끌어올렸다. 색상군을 통째로
바꾼 예외는 절대 없어야 한다 — 처음 고대비 모드를 만들 때 검정 배경에 노랑
액센트+라임 성공+핑크 위험+오렌지 경고를 무분별하게 섞었던 실패 사례가 있었고,
"브랜드와 무관한 OS 접근성 모드"처럼 보인다는 피드백으로 전부 브랜드 초록 계열로
재작업했다.

### 원칙 B — 절제된 상태색 (IBM Carbon 방식)

상태색은 success(초록) / danger(빨강) / warning(골드·앰버) 3색으로 제한한다.
새로운 상태가 필요해도 색을 더 늘리지 않고 아이콘·문구로 구분한다.

### 원칙 C — 다크 모드는 라이트의 반전이 아니라 별도 튜닝

- 배경은 순수 검정(`#000`)이 아니라 `#121212`~`#1e1e1e` 범위(눈부심·헤일레이션 방지,
  Material Design 다크 테마 가이드).
- accent는 라이트보다 채도를 낮추고 명도를 올린다(라이트 S48%→다크 S36%, 채도가
  높은 색은 어두운 배경 위에서 "진동"해 보여 가독성을 해친다).
- **상태색 3개(success/danger/warning)는 서로 명도가 비슷해야 한다.** 한 번은
  danger가 L82%로 success/warning(L65%)보다 유독 옅어서 "오류가 성공보다 시각적으로
  약해 보이는" 비대칭이 있었다 — red-300 대신 red-400을 써서 L71%로 맞췄다. 상태색을
  추가/수정할 때는 반드시 [contrast.js](#5-검증-방법)로 셋의 명도를 나란히 확인한다.

### 원칙 D — 고대비 모드(포용성 모드)

- AAA(7:1) 이상을 목표로 하는 고정 1종 룩. 배경은 `#0a0a0a`(완전한 `#000`이 아님 —
  대비비 손실은 사실상 없고 화면이 "뚫린 듯" 보이는 것만 방지).
- 구분선/테두리는 색이 아니라 **굵은 선**으로 표현한다(`[data-theme='high-contrast']
  .border { border-width: 2px; }`).
- 포커스 링은 기본 대비의 2배(`--focus-ring-width: 4px`).
- 그림자는 안 씀(`--shadow-modal: none`) — 어두운 배경 위 그림자는 잘 안 보이므로
  대신 테두리로 경계를 표현.
- **아이콘 단독 사용 금지**: 아이콘만으로 의미를 전달하는 버튼(전송, 닫기, 로그아웃,
  설정, 뒤로가기 등)은 고대비 모드에서 자동으로 텍스트 라벨이 함께 뜬다
  (`IconButton` 컴포넌트, `useTheme().resolvedTheme === 'high-contrast'`로 분기).
  상태 표시 아이콘(완료 체크 등)도 마찬가지로 라벨을 병기한다.

## 3. 색약 보정

드롭다운 3개: 없음 / 적록색약(protanopia+deuteranopia 통합, 내부 값 `red-green`) /
청황색약(tritanopia, 내부 값 `blue-yellow`).

### 왜 필터 방식을 안 쓰는가

색을 다루는 CSS/SVG filter(matrix) 방식은 대부분 "보정"이 아니라 **색약자가 보는
화면을 색약이 아닌 사람에게 시뮬레이션해서 보여주는** 필터(디자이너 테스트용)라
실제 색약 사용자 본인에게는 도움이 안 되거나 오히려 해가 될 수 있다. 대신 **상태색
전용 커스텀 토큰 세트**로 교체하는 방식을 쓴다.

### 색상 선정 기준 — Wong/Okabe-Ito 색맹 안전 팔레트

과학적으로 검증된 팔레트(Nature 등 학술지 표준)를 그대로 따른다: 파랑 `#0072B2`,
주황 `#E69F00`, 버밀리언 `#D55E00`, 노랑 `#F0E442` 등. 임의로 색을 골라잡지 않는다.

- **적록색약**: success→파랑, danger→버밀리언, warning→골드로 전면 교체. 초록·빨강
  둘 다 신뢰할 수 없으므로 완전히 다른 색상군을 쓴다.
- **청황색약**: 빨강↔초록 구분에는 영향이 없으므로(연구로 확인됨) success/danger는
  기본값 유지. 다만 원래 있던 "warning→분홍" 오버라이드는 **버그였다** — 분홍은
  노랑↔분홍 혼동쌍의 반대쪽 끝이라 danger의 빨강과 색상환에서 9~20도밖에 안
  떨어져 오히려 구분이 안 됐다. 이 앱의 기본 warning(버밀리언 계열, 순수 노랑이
  아님)은 애초에 파랑·순수노랑에 의존하지 않아 안전하므로, **불필요한 오버라이드는
  추가하지 않는 게 맞다** — 검증 없이 색을 바꾸는 것 자체가 위험하다는 교훈.

### 테마와 독립적으로 조합 가능해야 함

색약 보정은 라이트/다크/고대비 어떤 테마에서도 동작해야 한다(원래 요구사항).
`[data-theme='light'][data-colorblind='red-green']`처럼 두 속성을 조합한 선택자를
테마별로 각각 만든다 — "고대비는 명도 대비만으로 충분하니 색약 오버라이드는
필요 없다"고 가정하고 빠뜨렸던 적이 있는데, 저시력과 색각이상이 함께 있는
사용자를 고려하지 않은 잘못된 가정이었다. 새 테마나 새 색약 타입을 추가할 때
**모든 조합에 대해 실제로 CSS 선택자가 존재하는지 grep으로 확인**한다.

## 4. 글자 크기 / 굵은 텍스트 / 움직임 줄이기

### 글자 크기 (90%~200%)

- 메커니즘: `<html>`에 `--font-scale`을 인라인으로 세팅하고
  `html { font-size: calc(16px * var(--font-scale)) }`. Tailwind의 `text-*` 유틸리티는
  전부 rem 기반이라 자동으로 따라간다.
- **규칙: 실제 콘텐츠 텍스트에 `text-[15px]` 같은 px 고정값을 쓰지 않는다.**
  rem으로 쓴다(`text-[0.9375rem]`). px는 루트 font-size 변경에 반응하지 않는다 —
  채팅 말풍선과 시작 화면 본문이 이 실수로 실제로는 전혀 스케일되지 않던 버그가
  있었다. 예외는 `ThemePreviewThumb`처럼 **테마를 축소 미리보기하는 목업**뿐이다
  (여러 테마를 한 화면에 동시에 보여줘야 해서 실제 폰트 크기와 무관하게 고정
  크기를 유지해야 함).
- 상한은 WCAG 2.1 SC 1.4.4(Resize Text)가 요구하는 200%로 맞춘다. 범위를 넓힐 때는
  반드시 실제 컴포넌트(채팅/미션 카드/설정 화면 등)를 그 배율로 렌더링해서 레이아웃이
  깨지지 않는지 확인한 뒤 반영한다 — `truncate`가 걸린 요소는 우아하게 잘리는 것까지만
  허용, `whitespace-nowrap`으로 텍스트가 진짜 잘리는 건 금지.

### 굵은 텍스트

- 존재하는 모든 Tailwind font-weight 클래스(`font-medium/semibold/bold/extrabold`)를
  `[data-bold-text='true']` 아래에서 한 단계씩 올린다(400→500, 600→700, 700→800,
  800→900). **iOS/Android의 "굵게" 설정처럼 모든 텍스트를 하나의 굵기로 뭉개지
  않는다** — 그러면 제목과 본문의 위계가 사라진다(실제 앱들이 겪는 문제로 보고됨).
  단계적으로 올려 상대적 위계를 유지하는 게 이 앱의 방식이다.
- 아이콘(Material Symbols)은 `font-variation-settings`의 `wght` 축으로 굵기를
  별도 고정하므로 본문 굵기 오버라이드에 영향받지 않는다. 새 아이콘 컴포넌트를
  만들 때도 이 축을 명시적으로 고정할 것.
- 새 텍스트 요소를 추가할 때 Tailwind font-weight 클래스를 안 붙이면 `body`의
  상속값(400, 굵은 텍스트 켜지면 500)을 그대로 받는다 — 이것도 정상 동작이니
  일부러 `font-normal`을 붙일 필요는 없다.

### 움직임 줄이기

`[data-motion='reduced']`가 `<html>`에 있으면 전역적으로 모든 애니메이션/트랜지션
지속시간을 0에 가깝게 강제한다(`animation-duration: 0.001ms !important` 등). 새
애니메이션을 CSS keyframe/transition으로 추가하면 별도 처리 없이 자동으로 적용된다.

## 5. 검증 방법

색을 고르거나 바꿀 때는 감으로 판단하지 않고 항상 계산한다.

1. **대비비**: WCAG 상대휘도 공식으로 계산. 일반 텍스트 AA는 4.5:1, 큰 텍스트(24px
   이상 또는 18.66px 이상 굵게)는 3:1, 고대비 모드는 전부 7:1(AAA) 목표.
2. **명도(HSL Lightness) 균형**: 같은 역할을 하는 색끼리(예: success/danger/warning)
   HSL의 L값을 나란히 계산해서 한쪽만 유독 옅거나 진하지 않은지 확인.
3. **색상환 각도**: 두 색이 "다른 의미"를 전달해야 하는데(예: danger vs warning)
   Hue가 서로 30도 이내로 가까우면 실질적으로 구분이 안 된다고 보고 재검토.
4. **실제 렌더링**: Playwright로 헤드리스 브라우저에 실제 컴포넌트를 띄워
   라이트/다크/고대비 × 색약 보정 옵션의 모든 조합을 스크린샷으로 확인. 수치
   계산만 믿지 않고 항상 눈으로도 재확인한다.

## 6. 브랜드 컬러를 바꾸게 되면

`index.css` 상단 주석에도 체크리스트가 있다. 요약:

1. `tailwind.config.js`의 `brand` 색상 램프를 새 색으로 교체 (이 램프 자체는 어떤
   클래스명으로도 직접 쓰이지 않는 참고용 원본이다)
2. 라이트/다크/고대비 3개 테마 블록의 `--color-accent` 계열 4개 값만 새로 고르면
   된다 — `--color-success`와 `--color-focus-ring`은 `var(--color-accent...)`를
   참조하므로 자동으로 따라온다 (직접 새 값을 넣지 말 것)
3. `--color-highlight`(보조 강조색)는 accent와 독립적이라 자동으로 안 바뀜 — 새
   브랜드색과 어울리는지 디자인 판단 필요
4. `ThemePreviewThumb.jsx`의 `SWATCHES`는 3개 테마를 동시에 보여주는 미리보기라
   CSS 변수를 못 쓰고 하드코딩돼 있음 — 수동으로 맞춰야 함
5. 색약 보정 오버라이드가 새 브랜드색과 우연히 겹치지 않는지 재검토(예: 브랜드색이
   파랑 계열이 되면 적록색약 오버라이드의 "success=파랑"과 충돌 가능)
6. 카카오 로그인 버튼(`#FEE500`)은 브랜드 규정색이라 무관하게 그대로 둔다

## 7. 실제 참고 사례

색을 정할 때 감이 아니라 검증된 실제 사례를 먼저 찾아서 근거로 삼는다.

- **GitHub Dark High Contrast** — 7:1 이상을 확보하면서도 전체 다색 팔레트를 유지,
  텍스트 위계는 미세한 명도차로 유지
- **Material Design 3 톤 팔레트** — 대비 단계가 달라져도 같은 hue 유지, 명도만 조정
- **Windows 11 대비 테마(Aquatic 등)** — 순수 검정+원색이 아니라 짙은 회색+다중 톤의
  절제된 조합
- **IBM Carbon Design System** — 상태색은 빨강/주황·골드/초록 정도로 제한, 아이콘
  병행 필수
